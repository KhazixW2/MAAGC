# Panel 识别系统设计与实现

## 概述

本文档总结了在 MAAGC 项目中实现的面板（Panel）识别系统，包括潜力面板、血脉面板和特性面板的识别方案、技术选型和最佳实践。

## 一、系统架构

### 1.1 整体流程

```
父母信息识别 → 潜力面板 → 血脉面板 → 特性面板 → 子孙命名
```

### 1.2 核心组件

- **ChildRec Action**: 主控制器，协调整个识别流程
- **数据结构**: `Potential`, `Bloodline`, `Feature`, `ParentInfo`
- **配置管理**: `BLOODLINE_CONFIG`, `ATTRIBUTE_GROWTH_RANGES`
- **工具函数**: `get_attribute_grade()`, `generate_child_name()`

## 二、面板识别方案

### 2.1 父母信息面板

**识别目标**：父亲和母亲的姓名、爵位、佣兵团

**技术方案**：
- 使用固定的 ROI 区域识别
- OCR 识别后过滤置信度 > 0.5 的结果
- 爵位识别：检测"男爵"、"伯爵"、"公爵"、"骑士"关键词
- 佣兵团识别：ASCII 字符的短文本
- 姓名识别：2-10 个汉字的文本

**关键代码**：
```python
def extract_parent_info(self, context: Context, is_father: bool = True) -> ParentInfo:
    title_task = "PannelFatherTitle" if is_father else "PannelMotherTitle"
    reco_result = context.run_recognition(title_task, screenshot)
    # 解析爵位、姓名、佣兵团
```

**爵位比较**：
```python
title_rank = {"公爵": 4, "伯爵": 3, "男爵": 2, "骑士": 1, "无爵位": 0}
```

### 2.2 潜力面板

**识别目标**：六维属性（力量、体质、技巧、感知、敏捷、意志）及其数值

**技术方案**：
1. **锚点定位**：识别"潜力"文字作为锚点
2. **相对坐标**：使用预定义的 offset 表计算各属性位置
3. **动态 ROI**：
   ```python
   PanelPropertyTable = {
       "力量": {"attr_offset": [122, 58], "val_offset": [105, 88]},
       "体质": {"attr_offset": [360, 58], "val_offset": [369, 88]},
       # ...
   }
   ```
4. **属性识别**：
   - 属性名：匹配全称或第一个字（如 ["力量", "力"]）
   - 属性值：使用正则 `-?[0-9]+\.[0-9]+` 识别浮点数
5. **等级转换**：
   ```python
   ATTRIBUTE_GROWTH_RANGES = {
       "E": (-inf, 0.10), "D": (0.10, 0.20), "C": (0.20, 0.35),
       "B": (0.35, 0.55), "A": (0.55, 0.74), "S": (0.74, 0.93), "SS": (0.93, inf)
   }
   ```

**关键经验**：
- 使用 `roi` 而非 `crop_box`，更符合 MAA 标准
- 期望值使用数组，提高识别容错率
- 属性区间应该是连续的，避免空隙

### 2.3 血脉面板

**识别目标**：血统名称及其百分比

**技术方案**：
1. **双锚点定位**：
   - 血脉锚点（"血统"文字）
   - 特性锚点（"特性"文字）
   - 动态高度 = 特性锚点 Y - 血脉锚点 Y
2. **区域计算**：
   ```python
   BLOODLINE_CONFIG = {
       "name_offset_x": -10, "name_offset_y": -10, "name_width": 127,
       "percent_offset_x": 548, "percent_offset_y": 50, "percent_width": 73,
   }
   name_roi = [blood_x + offset_x, blood_y + offset_y, width, panel_height]
   ```
3. **分别识别**：
   - `PanelBloodNameCheck`：识别汉字（血统名）
   - `PanelBloodPercentCheck`：识别 `[0-9]+%`（百分比）
4. **一一对应**：利用 `order_by: Vertical` 排序，按索引匹配

**关键经验**：
- 使用双锚点计算动态高度，适配不同数量的血脉条目
- 将魔法数字提取为配置常量，便于后期维护
- 过滤"血统"标题文本，避免误识别

### 2.4 特性面板

**识别目标**：特性名称和描述

**技术方案**：
1. **锚点定位**：识别"特性"文字
2. **分阶段识别**：
   - 第一次：从特性锚点下方开始，高度 250px
   - 后续：使用固定 ROI `[21, 568, 686, 459]`
3. **循环下滑**：
   - 识别当前页特性
   - 检查是否有新特性
   - 连续 2 次未发现新特性则停止
   - 使用 `PropertyPanelSwipeDown` 下滑
4. **特性提取**：
   - 名称：2-10 个汉字，不包含"的、了、后"等
   - 描述：名称之后的文本
   - 跳过"隐藏特性"

**关键经验**：
- 第一次识别使用动态 ROI，避免误识别上方内容
- 滑动后使用固定大范围 ROI，覆盖所有可能区域
- 通过检查新特性数量判断是否继续滑动

## 三、子孙命名系统

### 3.1 命名规则

**格式**：`最高属性 + 次高属性 + 特性 + 爵位`

**示例**：
- `力 ss 技 ss 太科公`（力量 SS、技巧 SS、太初、科技、公爵）
- `感 s 意 a 神伯`（感知 S、意志 A、神秘、伯爵）

### 3.2 实现细节

1. **属性处理**：
   - 按属性值排序，找出最高和次高
   - 属性名取第一个汉字（如"力量"→"力"）
   - 等级转小写（如"SS"→"ss"）

2. **特性处理**：
   - 跳过"隐藏特性"
   - 取特性名第一个汉字
   - 最多取 3 个特性

3. **爵位处理**：
   - 取爵位第一个汉字（如"公爵"→"公"）

## 四、Pipeline 配置要点

### 4.1 OCR 识别配置

```json
{
  "PanelPropertyItemCheck": {
    "recognition": "OCR",
    "expected": ["力量", "力"],  // 支持多期望值
    "roi": [x, y, width, height],
    "order_by": "Vertical"  // 按垂直方向排序
  }
}
```

### 4.2 正则表达式

- 属性值：`-?[0-9]+\.[0-9]+`（匹配有符号浮点数）
- 血脉名：`[\u4e00-\u9fa5]+`（匹配汉字）
- 百分比：`[0-9]+%`（匹配数字+%）

### 4.3 滑动操作

```json
{
  "PropertyPanelSwipeDown": {
    "action": "Swipe",
    "begin": [364, 800, 1, 1],
    "end": [366, 640, 1, 1],
    "duration": 500,
    "post_delay": 500
  }
}
```

## 五、最佳实践总结

### 5.1 代码组织

1. **数据结构先行**：先定义 `Potential`、`Bloodline` 等结构体
2. **配置与逻辑分离**：魔法数字提取为常量
3. **函数职责单一**：每个函数只负责一个面板的识别
4. **状态管理**：使用类成员变量保存识别结果

### 5.2 识别优化

1. **锚点定位**：优先识别固定文字作为锚点
2. **相对坐标**：使用 offset 而非绝对坐标
3. **动态区域**：根据内容动态计算识别区域
4. **多重过滤**：置信度 + 文本内容 + 格式验证

### 5.3 容错处理

1. **多期望值**：支持识别全称或简称
2. **连续失败判断**：连续 2 次无新内容才停止
3. **默认值处理**：未识别到爵位时设为"无爵位"
4. **日志精简**：只输出关键信息，便于调试

### 5.4 性能考虑

1. **减少截图次数**：一次截图用于多个识别
2. **ROI 精确化**：缩小识别区域提高速度
3. **提前终止**：检测到目标后立即停止

## 六、常见问题与解决方案

### 6.1 识别不准确

**问题**：OCR 识别结果包含噪声

**解决**：
- 提高置信度阈值（> 0.9）
- 使用正则过滤格式
- 关键词排除（如"的"、"了"）

### 6.2 区域不匹配

**问题**：固定 ROI 无法适配不同情况

**解决**：
- 使用双锚点计算动态高度
- 第一次使用动态 ROI，后续使用固定 ROI

### 6.3 类型定义错误

**问题**：`NameError: name 'Potential' is not defined`

**解决**：
- 避免在模块级别函数中使用未定义的类作为类型注解
- 使用字符串注解或移除类型注解

## 七、后续优化方向

1. **配置文件化**：将 `BLOODLINE_CONFIG` 等移到 config.ini
2. **更多爵位支持**：扩展爵位等级体系
